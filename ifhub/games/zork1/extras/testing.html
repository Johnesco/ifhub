Testing

How we verify every version of Zork I

Zork I is a 350-point adventure game with random combat, a roaming thief, multiple timer systems, and dozens of puzzle sequences that must work in exact order. A single broken rule can silently block the player from winning. Our testing approach uses deterministic RNG seeding, automated walkthrough verification, and targeted regression tests to catch problems before they ship.

The Problem: Randomness

Zork I uses a random number generator (RNG) for combat outcomes, thief movement, and several other systems. This means the same sequence of commands can produce different results on each playthrough. The troll might die in 3 hits or 12. The thief might steal a critical treasure before you reach it, or never appear at all.

To test reliably, we need to control the RNG. Both interpreters we use support seed-based determinism: given the same seed, the same sequence of inputs always produces the same outputs. This transforms a nondeterministic game into a reproducible, testable system.

Software Stack

All testing runs under WSL (Windows Subsystem for Linux) to use native Linux builds of the interpreters, which support RNG seeding via command-line flags.

Component Tool Purpose
I7 Compiler inform7.exe + inform6.exe Compiles Inform 7 source to Glulx bytecode (.ulx)
Glulx Interpreter glulxe (CheapGlk build) Runs .ulx games headlessly with --rngseed N
Z-machine Interpreter dfrotz Runs .z3 games (ZIL version) with -s N
Regression Framework RegTest (Python) Pattern-matching test runner for interactive fiction
Shell Scripts run-walkthrough.sh, find-seeds.sh, run-tests.sh Automation wrappers with diagnostics

Test Types

1. Walkthrough Verification (Primary)

The most important test: a 439-command walkthrough that completes the entire game, collecting all 19 treasures and reaching the endgame with a perfect score of 350/350. This is the same walkthrough used for both the ZIL original and every Inform 7 version.

The walkthrough is piped directly into the interpreter via stdin. The test runner then parses the output transcript for:

  • Final score — must be 350/350
  • Death count — must be 0 (with correct seed)
  • Troll status — survived combat or killed
  • Thief encounters — number of appearances and thefts
  • Parser errors — "can't see any such thing" or "can't go that way" counts
  • Score changes — number of increases and decreases
  • Endgame reached — confirmation the game was won
$ wsl -e bash tests/run-walkthrough.sh --seed 30

=== Zork I Walkthrough Test ===
Engine:  glulxe (inform7)
Seed:    30
Score:   350/350  ✓
Deaths:  0
Troll:   SURVIVED (1 blocking messages)
Thief:   11 mentions, 0 thefts detected
Errors:  17 "can't see" / 0 "can't go" / 0 other
Scoring: 45 increases, 2 decreases
Endgame: REACHED
Result:  PASS

2. Golden Seed System

A golden seed is an RNG seed value proven to produce a 350/350 walkthrough with zero deaths. Golden seeds are stored in seeds.conf along with a hash of the compiled binary they were validated against: 

# seeds.conf
# Format: engine:seed:binary_hash:date
glulxe:30:7c49db5f:2026-02-20
dfrotz:3:2cfd8723:2026-02-18

When the compiled binary changes (new code), the hash check warns that the golden seed may no longer work. If it doesn't, find-seeds.sh sweeps seeds 1 through 200 to find a new one: 

$ wsl -e bash tests/find-seeds.sh

=== Zork I Seed Sweep ===
Mode:    Inform 7 (glulxe)
Range:   1..200

[seed 30/200] *** 350/350 PASS *** (seed 30)

=== Golden Seed Found! ===
Engine:  glulxe
Seed:    30
Score:   350/350
Hash:    a1b2c3d4

Why seeds break

Adding new objects, rules, or text to the Inform 7 source can shift the internal RNG sequence. A seed that produced favorable combat rolls before the change may produce unfavorable ones after. This doesn't mean the game is broken — it means the specific seed no longer aligns with the walkthrough's combat requirements. A new seed almost always exists within the first 200 values.

3. RegTest Regression Tests

RegTest is Andrew Plotkin's pattern-matching test framework for interactive fiction. It sends commands to the interpreter and checks the output against expected patterns (regex or literal strings).

Our test suite covers key game paths:

  • startup — Initial room description and mailbox
  • enter-house — Kitchen window entry
  • living-room — Gear pickup (sword, lantern)
  • cellar — Rug, trap door, cellar entry
  • troll — Combat sequence (RNG-dependent)
  • echo — Loud Room echo puzzle
  • dam — Power button, wrench, sluice gates
  • mirror — Mirror room teleportation
  • cyclops-odysseus — Cyclops puzzle (word solution)
  • forest-loop — Forest navigation consistency
  • house-circuit — Exterior house traversal
$ wsl -e bash tests/run-tests.sh

RegTest: running 14 tests...
  startup: passed
  mailbox: passed
  enter-house: passed
  ...
All tests passed.

4. Built-in Test Commands (v3)

Version 3 includes Inform 7's built-in Test commands, which are macros that execute a sequence of game commands in a single action. These are available during play for quick smoke-testing of specific puzzle sequences: 

> test cellar
(Testing with: push rug / open trap door / down ...)

> test exorcism
(Testing with: se / e / tie rope to railing / d / take torch ...)

Available test macros: cellar, troll, cyclops, dam, exorcism, machine, boat.

Workflow

Every code change follows this sequence before being committed:

  1. Edit — Modify the Inform 7 source (story.ni) in the hub at C:\code\inform7\projects\zork1\.
  2. Compile — Two-stage compilation: Inform 7 to Inform 6 (inform7.exe), then Inform 6 to Glulx bytecode (inform6.exe). Compilation errors are caught here.
  3. Walkthrough test — Run run-walkthrough.sh --seed 30. Must achieve 350/350 with zero deaths. This is the gate: no commit happens unless the walkthrough passes.
  4. Propagate — The hub source is propagated to all three versioned snapshots (v1, v2, v3), each compiled and tested independently. Version differences are created by surgically removing version-specific code:
    • v3 = full hub source (test commands included)
    • v2 = hub minus Chapter 16 (test commands)
    • v1 = v2 minus 7 specific bug-fix blocks (faithful port, no fixes)
  5. Deploy — Each version's compiled .ulx is base64-encoded into zork1.ulx.js for the web player. Pushed to GitHub, deployed automatically via GitHub Pages.

Three versions, one source

All three Inform 7 versions (v1, v2, v3) are derived from a single hub source file. Version differentiation happens by removing specific code blocks during propagation, not by maintaining separate source trees. This ensures that text corrections and new features automatically appear in all versions where they belong.

Cross-Version Verification

The ZIL original (v0) serves as the behavioral reference. The same 439-command walkthrough is run against both the ZIL binary (via dfrotz) and each Inform 7 binary (via glulxe). Both must achieve 350/350.

Version Engine Golden Seed Score
v0 (ZIL) dfrotz 3 350/350
v1 (Faithful Port) glulxe 30 350/350
v2 (Bug Fixes) glulxe 30 350/350
v3 (Audio & Testing) glulxe 30 350/350

The ZIL and Inform 7 versions use different golden seeds because they have different RNG implementations. The walkthrough commands are identical; only the seed differs.

What the Tests Catch

  • Broken navigation — A missing room connection causes the walkthrough to stall at "You can't go that way"
  • Missing objects — "I can't see any such thing" errors spike when an object isn't where expected
  • Scoring bugs — Score count diverges from expected 45 increases and 2 decreases
  • Timer regressions — Lamp dying too early, candles burning out at wrong time, match count off
  • Combat regressions — Troll or thief combat rules changing behavior under the golden seed
  • Puzzle sequence breaks — Dam not opening, exorcism not completing, machine not transforming coal
  • Death loops — Unexpected deaths show up in the death counter (should be 0)

What the Tests Don't Catch

Automated walkthrough testing verifies the critical path — the exact sequence of commands needed to win. It does not cover:

  • Alternate solutions — Feeding the cyclops lunch + water (vs. saying "odysseus")
  • Edge-case interactions — Putting unexpected items in unusual containers
  • Text accuracy — Whether a response matches the original ZIL wording (requires manual ZIL-to-I7 audits)
  • Off-path exploration — Rooms and objects the walkthrough never visits
  • Parser edge cases — Unusual phrasings, ambiguous nouns, multi-word commands

RegTest partially addresses these gaps by testing specific interactions independent of the walkthrough, but full coverage of a game this size requires ongoing manual review against the ZIL source.
