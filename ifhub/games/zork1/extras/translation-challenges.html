Translation Challenges

ZIL to Inform 7 — what looked straightforward wasn’t

Translating Zork I from ZIL to Inform 7 looked like a straightforward port. Both languages describe rooms, objects, and rules. The map is the same. The puzzles are the same. How hard could it be?

It turned into an archaeology project. ZIL and Inform 7 don’t just have different syntax — they have different assumptions about how a world model works. Containment, scope, action routing, property namespaces, text output, and NPC behavior all operate under fundamentally different rules. The game compiled. It ran. It looked right. And then testing revealed that half of it was quietly broken.

These are the six biggest surprises — the problems that took the most time to find, understand, and fix.

1. The Leaf Count: 377 Missing Text Responses

ZIL routines are built from nested COND trees — cascading conditional branches where each terminal node (or “leaf”) produces a different text response via TELL. A single routine like V-FIND might have eight or more conditional branches, each printing unique text depending on context: what the player is holding, where they are, whether an object is visible, and so on.

The initial translation handled the main paths through each routine — the ones the walkthrough exercised. But ZIL’s branching structures have many alternate leaves for edge cases, unusual inputs, and specific object combinations. These are easy to miss because the code compiles fine without them; you just get silence where the original game would have said something.

Systematically counting every leaf in every ZIL routine revealed 377 missing text responses. The game was playable and winnable, but it was only saying about half of what it should. A player who tried anything off the critical path would get no response instead of the original’s carefully written text.

Restoring them required a 6-phase audit:

Phase Scope Issue
1. Game systems Scoring, death, movement, inventory, lamp, combat framework #45
2. Rooms & objects Room descriptions, object first-descriptions, container text #46
3. Object interactions Taking, dropping, opening, closing, inserting, removing #47
4. Verb handlers Read, find, pour, fill, dig, climb, wave, burn, and 30+ others #48
5. Scenery Trees, water, walls, doors, stairs, and other examinable scenery #52
6. Combat messages 88 distinct melee messages across 7 outcome categories #58

Why this was hard to catch

A missing text response doesn’t cause an error. The game just says nothing. During normal play, you might assume the parser didn’t understand your input. Only by reading the ZIL source line by line and counting every TELL statement could we confirm what was supposed to be there.

2. The Parser Conflict: Reading vs. Examining

Zork I has a read command. You read the matchbook. You read the engravings. And in the exorcism ritual, you read a black book to banish the spirits of Hades — it’s how you trigger the climactic “Begone, fiends!” moment.

Inform 7’s Standard Rules define read as a synonym for examining. This is hardcoded in the library. When the game defines a custom reading action for objects like the book, the parser has two competing interpretations of the word “read.”

It silently chose the built-in one. READ BOOK triggered examining, not reading. The player saw the book’s description (“Each page contains a Commandment”) instead of the exorcism text (“Begone, fiends!”). The ritual never completed. The game couldn’t be won through that puzzle path. 

ZIL:
> read book
The prayer seems to protect you as the spirits in this room change
their form. As the words of the prayer continue, the spirits fade
and the room is filled with bright light. "Begone, fiends!" ...

I7 (before fix):
> read book
This book is large and bound in leather. Each page contains a single
Commandment of the Great Underground Empire.

The fix required intercepting the parser before it could route to the built-in action, explicitly redirecting read [something] to the custom reading action when a readable object is involved. (#67)

3. Containment Semantics: Where Is the Player?

Inform 7’s containment model is hierarchical: the player is in a boat, the boat is in a room. This seems natural. But the player is in the River checks direct containment only — it returns false when the player is in a boat that’s in the River. The player is in the boat, not in the room.

This broke river navigation, the launch command, and any rule that checked the player’s location by room rather than by containment chain. The fix was to use the location of the player (which walks up the containment tree) instead of direct is in checks.

A related problem: move the player to X is Inform 7’s low-level teleportation. It bypasses After rules entirely. The Treasure Room awards a +25 bonus on first visit via an After rule — but because the player was moved there rather than going there, the After rule never fired. The bonus was never awarded. 

Broken:
  if the player is in the Frigid River: [false when in boat!]
  move the player to the Treasure Room; [bypasses After rules!]

Fixed:
  if the location of the player is the Frigid River: [works]
  now the player is in the Treasure Room; [fires After rules]

These two patterns — direct containment checks and low-level movement — affected river navigation (#15), the launch command, visit bonuses (#28), and any rule testing the player’s room when a vehicle was involved.

The subtlety

ZIL doesn’t have this distinction. LOC in ZIL always returns the room, regardless of intermediate containers. The translation looked correct because the Inform 7 code read naturally — “if the player is in the River” sounds right. It just doesn’t mean what you think it means.

4. Property Shadowing: The Invisible Problem

ZIL uses a FSET/FCLEAR system for object flags. Objects can be INVISIBLE (hidden from room listings) or have the flag cleared to become visible. The Inform 7 translation defined custom properties: 

A thing can be visible or invisible.

This compiled without error. It appeared to work. But “visible” is a word Inform 7 uses internally for its scope system — the mechanism that determines which objects the player can see and interact with. Defining a custom property with the same name shadowed the built-in concept.

The result: items that were supposed to be hidden from room listings appeared anyway. The custom invisible flag was being set, but the engine’s own scope system didn’t respect it because it was checking a different “visible.” There was no error, no warning. The property just silently didn’t do what it was supposed to.

The fix was straightforward once diagnosed: rename to zil-visible and zil-invisible to avoid the namespace collision. But diagnosing it took much longer than fixing it — the symptom (extra items in room listings) didn’t obviously point to a property name conflict. (#63)

The lesson

Inform 7’s natural-language syntax makes it easy to accidentally use a word the engine already claims. There is no reserved-word list for properties. If your custom property name overlaps an internal concept, the compiler won’t tell you. You’ll find out at runtime, if you’re lucky.

5. FDESC and Container Visibility

In ZIL, objects inside containers can have an FDESC (first description) — rich text that prints automatically when the player first encounters the container’s room. The jewel-encrusted egg is the iconic example: 

ZIL (on entering the Bird Room for the first time):
In the bird's nest is a large egg encrusted with precious jewels,
apparently scavenged by a childless songbird. The egg is covered
with fine gold inlay, and ornamented with lapis lazuli and
mother-of-pearl.

Inform 7 doesn’t do this. When describing a room, it mentions containers and what’s inside them, but it doesn’t print the contained object’s own description text. The egg’s lovingly written first description was invisible — the player just saw “In the bird’s nest is a jewel-encrusted egg.”

The same pattern affected every object introduced via its container:

  • The jewel-encrusted egg in the bird’s nest (#79)
  • The sceptre in the coffin (#80)
  • The canary in the egg (#81)
  • Thief treasure container contents (#82)

For a detailed analysis of this problem and the solutions used, see FDESC: First Descriptions in Containers.

Each required a custom “after examining the container” rule or a writing-paragraph-about rule to inject the contained object’s first description into the room text at the right moment. It’s one of those cases where ZIL’s simpler model (just print the text) was actually more flexible than Inform 7’s more structured approach.

6. The Ghost That Wasn’t There: NPCs and State Machines

Zork I has three NPCs: a troll guarding a bridge, a thief who roams the dungeon stealing treasures, and a cyclops blocking a passage. Each has a multi-state behavior model in ZIL — the troll can be conscious, unconscious, or dead; the thief patrols on a timer and decides whether to steal, fight, or flee; the cyclops responds to feeding, watering, and specific words.

The thief

The thief is the most complex NPC in the game. Every 5 turns, he evaluates whether to enter the player’s room, steal a valuable item, engage in combat, or move on. He redistributes stolen treasures to random rooms. He’s repelled by garlic. He has his own combat table with unique messages.

After the initial translation, the thief “appeared to work” — he was defined, he had rules, the game compiled. But he never actually roamed. His every-turn patrol rule wasn’t firing because of how Inform 7 schedules timed events versus ZIL’s turn-based routine calls. The thief sat in his lair for the entire game, a ghost NPC. (#7, #8)

The troll

The troll’s state machine has subtle transitions: unconscious trolls recover after a set number of turns, dead trolls stay dead, and the troll’s weapon status affects combat. Flag bugs in the translation meant the troll could get stuck in impossible states — unconscious but still fighting, or dead but blocking the bridge. (#6, #59)

Combat messages

ZIL’s combat system uses 88 distinct text messages across 7 outcome categories: attacker hits, attacker misses, defender staggers, weapon knocked away, light wound, serious wound, and killing blow. Each category has multiple variants selected randomly for variety. These were among the 377 missing text responses discovered in the leaf count audit. (#56, #58)

Why NPCs are the hardest part

Room descriptions and puzzle mechanics are static — they either work or they don’t. NPCs are dynamic: their behavior depends on game state, turn counters, random rolls, and interactions between multiple systems. A broken room is immediately obvious. A broken NPC can silently fail for an entire playthrough, and you won’t notice unless you know exactly what it was supposed to do.

Closing Thoughts

What looked like a translation turned out to be an archaeology project. Understanding not just what the ZIL code does, but what it meant to do, required reading every routine, counting every branch, and testing every edge case. The two languages agree on the surface — rooms, objects, rules — but disagree on almost everything underneath.
