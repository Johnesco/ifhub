1. Overview

IF Hub is a static site that hosts multiple Inform 7 games in a unified browser interface. Games are written in Inform 7 (a natural-language programming language), compiled to virtual machine bytecode, base64-encoded into JavaScript, and played in the browser via Parchment (a Glulx/Z-machine interpreter).

There is no server-side logic. Everything — source code, compiled binaries, the interpreter, sound files — is static HTML, CSS, and JS served from a single directory.

story.ni → story.i6 → game.ulx → game.ulx.js → play.html (Inform 7) (Inform 6) (Glulx VM) (base64 JS) (browser) With sound: story.ni → game.ulx → game.gblorb → game.gblorb.js → play.html (Inform 7) (Glulx VM) (blorb+audio) (base64 JS) (browser)

3. Compilation Pipeline

Standard compile (no sound)

bash tools/compile.sh sample
  1. Inform 7 → Inform 6 — inform7.exe translates the natural-language story.ni into Inform 6 intermediate code (story.i6)
  2. Inform 6 → Glulx — inform6.exe compiles story.i6 into a .ulx Glulx virtual machine binary
  3. Clean up — removes the intermediate story.i6
  4. Web player setup — setup-web.sh copies Parchment libraries, base64-encodes the .ulx into a .ulx.js file, and generates play.html

Compile with sound

bash tools/compile.sh zork1 --sound

Same first two steps, then:

  1. Generate blurb — generate-blurb.sh parses story.ni for Sound of X is the file "Y.ogg" declarations, creates a .blurb manifest mapping resource IDs (3, 4, 5…) to .ogg files in Sounds/
  2. Build blorb — inblorb.exe reads the .blurb and packages the .ulx + all .ogg files into a single .gblorb (Glulx blorb) container
  3. Clean up — removes story.i6 and .blurb
  4. Web player setup — base64-encodes the .gblorb (not the .ulx) so the browser gets the full blorb with embedded audio

Pre-flight checks

Before starting the expensive compilation, compile.sh validates:

  • No colons in the story title — Windows can’t create files with : in the name, and Inform 7 derives filenames from the title. Use - instead.
  • Sounds/ directory exists (when --sound) — the .ogg files must be at the project root, not inside .materials/Sounds/.

4. Sound (Native Blorb)

Games with sound use native Glk/Blorb audio. The .ogg files are embedded directly inside the .gblorb binary. When the game issues Glk sound channel calls, Parchment decodes and plays them via the browser’s AudioContext. No separate audio files or JavaScript overlay needed.

How it works

story.ni declares: Sound of forest is the file "forest.ogg" Sound of sword is the file "sword.ogg" generate-blurb.sh creates .blurb: storyfile "game.ulx" include sound 3 "Sounds/forest.ogg" sound 4 "Sounds/sword.ogg" inblorb packages: game.ulx + forest.ogg + sword.ogg → game.gblorb base64 encodes: game.gblorb → game.gblorb.js Browser: parchment.js loads gblorb.js → parses blorb resource map → extracts audio chunks → AudioContext decodes + plays on demand

Requirements

  • Sound declarations in story.ni: Sound of X is the file "Y.ogg"
  • .ogg files in projects/<name>/Sounds/
  • Compile with --sound flag
  • Play page must load parchment.js (not main.js)

5. Web Player (Parchment)

Parchment 2025.1 is the browser-side interpreter. It ships as 12 files:

  • parchment.js — main engine with AudioContext sound support
  • parchment.css — engine styling
  • main.js — AsyncGlk standalone build (no sound — do not load for sound games)
  • main.css — layout styling
  • jquery.min.js — jQuery
  • quixe.js — Quixe (JavaScript Glulx interpreter)
  • glulxe.js — Glulxe (WASM Glulx interpreter)
  • bocfel.js — Bocfel (Z-machine interpreter)
  • zvm.js — ZVM (Z-machine interpreter)
  • ie.js — IE compatibility shim
  • resourcemap.js — blorb resource mapping
  • waiting.gif — loading animation

Each play page defines a parchment_options object that tells Parchment where to find the game binary, the library files, and the story name: 

parchment_options = {
    default_story: ['path/to/game.gblorb.js'],
    lib_path: 'lib/parchment/',
    story_name: 'game.gblorb.js',    // REQUIRED
    use_proxy: 0,
    do_vm_autosave: 1,
};
story_name is required. parchment.js calls .substring() on it to detect the file type. If missing, you get TypeError: Cannot read properties of undefined (reading 'substring').

6. Hub Deployment

The hub at ifhub/ aggregates all games into a single landing page. Deployment is a two-step process:

Step 1: Compile each game

bash tools/compile.sh zork1 --sound
bash tools/compile.sh feverdream --sound
bash tools/compile.sh dracula
bash tools/compile.sh sample

Step 2: Publish each game

Each game project deploys to GitHub Pages from its own repo. The hub references games by URL — no copying or deployment needed.

Adding a new game

Update games.json and cards.json:

  1. Ensure the game has the 4 standard pages (play.html, source.html, walkthrough.html, index.html)
  2. Deploy the game repo to GitHub Pages
  3. Add a game entry to games.json with URL fields (playUrl, sourceUrl, walkthroughUrl, landingUrl)
  4. Add card metadata to cards.json

Entry points

Each game is playable from two places:

  • Game’s own site — e.g., /zork1/v4/play.html — full-page player served from the game’s own GitHub Pages repo
  • IF Hub app — app.html?game=<id> — split-pane interface with source viewer; iframes the game’s own play page

7. Testing

Tests run in WSL using native Glulx/Z-machine interpreters (not the browser). Each project has a tests/ directory with:

  • project.conf — engine paths, score patterns, golden seeds
  • run-tests.sh — RegTest regression tests
  • run-walkthrough.sh — full walkthrough with score checking
  • find-seeds.sh — sweep RNG seeds for deterministic runs
# Run regression tests
cd projects/zork1 && wsl -e bash tests/run-tests.sh

# Run walkthrough
cd projects/zork1 && wsl -e bash tests/run-walkthrough.sh

# Find golden seeds after code changes
cd projects/zork1 && wsl -e bash tests/find-seeds.sh

8. Known Gotchas

All of these are guarded by the scripts, but good to know:

parchment.js vs main.js

Parchment 2025.1 ships two JavaScript entry points. parchment.js (134 KB) has real AudioContext sound channel support. main.js (176 KB) is the AsyncGlk standalone build with stub sound functions that throw errors and a hardcoded gestalt_Sound = 0. If a play page loads main.js, the game silently disables sound and prints [Sound effect number N here.] as text fallback. Always load parchment.js.

Colon in story title

Inform 7 derives filenames from the title on line 1 of story.ni. A colon (:) is illegal in Windows filenames. "Zork I: The Great" would produce Zork I: The Great Underground E.gblorb which can’t be written. Use a dash instead: "Zork I - The Great".

Sounds/ directory location

compile.sh --sound looks for .ogg files at projects/<name>/Sounds/ (project root). If your sounds are in the Inform 7 IDE convention (<name>.materials/Sounds/), copy them over. Do not symlink — sound directories may diverge between projects.

Missing story_name

parchment.js calls .substring() on parchment_options.story_name to detect the file type. If it’s missing, you get a TypeError crash. Every play page must include story_name.

Browser cache

After rebuilding, browsers may serve stale JavaScript. Both setup-web.sh and setup-web.sh append ?v=<timestamp> cache-busting params to all .js and .css references. If testing manually, use Ctrl+Shift+R or an incognito window.

Upstream Inform 7 bugs (v10.1.2)

Two bugs in the Inform 7 IDE’s Release pipeline prevent native blorb sound from working out of the box:

  1. The generated Release.blurb puts colons from the title into the storyfile leafname, creating invalid filenames on Windows.
  2. The base64 line encodes the .ulx (naked binary) instead of the .gblorb (blorb with sounds), so the web player gets no audio data.

Both are worked around by compile.sh --sound, which generates the blurb manually and encodes the .gblorb.
